Your feedback.txt outlines four pipeline stages: Trial (generate demo versions), Role (define audiences), Tasks (user scenarios to test against), and Evaluator (assess and rank the versions). This is essentially a harness engineering workflow for iteratively improving your dashboard demo.

Here's what I'd recommend doing before you create any of the agents/skills:

1. Lock down the data foundation first. Your agents will be generating and evaluating dashboard versions against your biomechanics data. Before they can do that reliably, the CSV, the data dictionary, and the metric labels need to be fully consistent and stable. Right now you already have good documentation, but I'd suggest a quick audit pass — confirm the CSV column count matches the data dictionary (you have 155 columns in the CSV but the dictionary documents 141), verify the registration columns are all documented, and ensure metric_labels.json is complete. If an agent generates a dashboard variant that references a metric not in the labels file, things break silently.

2. Define your audience personas concretely. Your feedback says "researchers, clinicians, patients ??" — that question mark is critical. Each audience has very different needs: a clinician wants quick red-flag triage and comparison to thresholds, a researcher wants filtering and statistical summaries, and a patient wants plain-language recovery trajectory. You need to decide which audiences are in scope for the demo, and write a short persona document for each one (what they care about, what they'd click on, what language level they expect). This becomes the input for the "Role" agent.

3. Write out your test scenarios before building the Tasks agent. The Tasks agent needs concrete workflows to test against — not just vague goals like "easy to understand." I'd suggest writing 5–8 specific scenarios, something like: "A physiotherapist wants to show a 25-year-old male ACLR patient at 6 months post-op how his single-leg drop jump RSI compares to similar patients" or "A researcher filters for all female basketball players and exports population statistics." These become your ground-truth test cases. Without them, the evaluator has nothing objective to measure against.

4. Define your evaluation criteria and scoring rubric. The Evaluator agent needs to know what "best" means. Is it clinical accuracy? Speed of finding information? Visual clarity? I'd suggest defining 4–6 criteria (e.g., clinical correctness, workflow efficiency, visual clarity, data accessibility, appropriate use of red-flag thresholds) and a simple scoring scale for each. This way the evaluator produces comparable, quantitative output rather than subjective prose.

5. Version the current prototype as your baseline. Before the Trial agent starts generating variants, save the current app.py and index.html as an explicit "v0 baseline." Every variant the Trial agent produces should be a diff from this baseline, so the Evaluator can compare apples to apples.

In short: clean data → defined personas → concrete test scenarios → scoring rubric → versioned baseline. Once those five artefacts exist, each of your four agents has clear inputs and constraints, and the pipeline can run without ambiguity.

There are actually two different "skills" locations in play, and it's important to understand the difference:

1. /mnt/.claude/skills/ — This is where Cowork's built-in skills live (docx, pdf, pptx, xlsx, etc.). These are read-only system skills. You can't and shouldn't modify these.

2. /mnt/OSVi_database_demo/skills/ — This is inside your project folder. You've already been using it for your project-level SKILL.md and data dictionary. This is your space to work with.

For your harness engineering pipeline, I'd recommend organising your project's skills/ folder to mirror your four pipeline stages, while keeping common reference material at the top level. Something like this:

OSVi_database_demo/
├── skills/
│   ├── SKILL.md                          ← master project skill (already exists)
│   ├── OSVi_Biomechanics_Demo_Database_DataDictionary.md  ← (already exists)
│   │
│   ├── harness/                          ← harness pipeline folder
│   │   ├── 1_trial/
│   │   │   └── SKILL.md                  ← instructions for generating demo variants
│   │   ├── 2_role/
│   │   │   └── SKILL.md                  ← audience personas (clinician, researcher, etc.)
│   │   ├── 3_tasks/
│   │   │   └── SKILL.md                  ← user scenarios & test workflows
│   │   └── 4_evaluator/
│   │       └── SKILL.md                  ← scoring rubric & evaluation criteria
│   │
│   └── shared/                           ← shared artefacts all agents reference
│       ├── personas.md                   ← detailed audience persona definitions
│       ├── test_scenarios.md             ← the 5–8 concrete test cases
│       └── scoring_rubric.md             ← evaluation criteria & scale

Why this structure works:

Each numbered folder under harness/ becomes a self-contained skill with its own SKILL.md. When you invoke a step, that agent reads its own SKILL.md for instructions, plus it can reference the shared/ folder for common context like personas and test scenarios. The numbering makes the execution order explicit.

The shared/ folder is key — it prevents duplication. The Trial agent needs to know the personas (to generate appropriate variants), the Tasks agent needs them too (to write scenarios from each persona's perspective), and the Evaluator references them when scoring. Rather than repeating this in every SKILL.md, they all point to shared/personas.md.

Your existing top-level SKILL.md stays as the master project context — it's already comprehensive and every agent should read it first before doing anything else.

What you don't need to do: You don't need to create anything under /.claude/skills/ — that's the system area. All your custom skills live in your project folder.

Claude

You're on the right track, but I'd refine it a bit.

Tasks — yes, but more structured than just a list. You're right that it's about user scenarios, but each scenario needs to be specific enough that an agent can actually execute it against a dashboard variant and determine pass/fail. So rather than "clinician checks patient recovery," you'd want something like: "A physiotherapist filters for male, ACLR, basketball, 6 months post-op, then looks at single-leg drop jump RSI LSI, and expects to see a red flag if it's below 80%." Each scenario should specify who the user is, what they do step by step, and what the expected outcome looks like. Think of them as test cases, not just descriptions.

Evaluator — yes, but it needs a scoring rubric, not just a list of factors. Listing factors like "clarity" and "usefulness" is a start, but the agent won't produce consistent results unless each factor has a defined scale. For example: "Clinical accuracy — 1: incorrect thresholds or labels, 2: mostly correct but missing edge cases, 3: all thresholds and labels match the data dictionary." Without that scale, the evaluator will just write vague opinions each time it runs, and you can't meaningfully compare versions.

So to correct slightly:

Tasks/SKILL.md should contain the scenarios plus instructions telling the agent how to turn those scenarios into structured test runs — what to check, how to record results, what format to output.

Evaluator/SKILL.md should contain the factors plus the scoring rubric plus instructions on how to aggregate scores across scenarios and produce a final ranking of the demo versions.

The SKILL.md in each folder isn't just data — it's the agent's instruction manual. The scenarios and rubric can live as separate files alongside it (like test_scenarios.md and scoring_rubric.md), and the SKILL.md tells the agent how to use them.
